# CUB-200实验指令文档

本文档按“当前代码已经可以直接运行”的口径，整理 CUB-200 单域分类模式下的实验命令。

当前可运行链路为：

1. 配置并激活环境
2. 下载并整理 CUB-200 数据集
3. 检查 Refool 反射图目录
4. 进行 1 epoch smoke test
5. 训练干净模型（benign baseline）
6. 训练 4 种攻击模型：`BadNets / Blended / WaNet / Refool`
7. 针对每种攻击训练 `REFINE`
8. 评估 `BA` 和 `ASR_NoTarget`
9. 汇总最终结果

需要特别说明：

- 本文档针对的是 **CUB-200 单域分类实验**。
- 当前仓库还 **没有** PACS 接入与跨域实验逻辑，因此本文档不包含 PACS 命令。
- 推荐优先使用统一入口脚本，而不是旧的 GTSRB 专用薄包装脚本。

## 1. 脚本入口约定

推荐统一入口：

- `scripts/bootstrap_datasets.py`
- `scripts/prepare_cub200_dataset.py`
- `scripts/train_benign.py`
- `scripts/train_attack.py`
- `scripts/train_refine.py`
- `scripts/eval_refine.py`
- `scripts/collect_final_results.py`

这些脚本都已经支持：

- `--dataset cub200`

## 2. 运行目录

所有命令默认都在仓库根目录执行：

- Windows 本地：`C:\Users\17672\Documents\Projects\SBBiShe`
- Linux 服务器：`/root/SBBiShe`

下面的命令默认以 Linux 服务器为主；如果在本地执行，只需要把路径替换成对应本地路径即可。

## 3. 环境配置与激活

### 3.1 推荐做法：使用独立 conda 环境

如果环境还没创建，可以先执行：

```bash
cd /root/SBBiShe
conda create -n sbbishe python=3.10 -y
conda activate sbbishe
pip install -r requirements.txt
pip install torch==2.1.2 torchvision==0.16.2 --index-url https://download.pytorch.org/whl/cu121
```

如果环境已经存在，则只需要激活：

```bash
cd /root/SBBiShe
conda activate sbbishe
```

说明：

- `requirements.txt` 当前包含项目运行所需的常用依赖。
- `torch / torchvision` 建议单独安装。
- CUB-200 图像尺寸默认为 `128x128`，显存占用会高于 GTSRB，批大小建议比 GTSRB 更保守。
- 如果环境里设置了 `OMP_NUM_THREADS` 或 `MKL_NUM_THREADS`，请确保它们是正整数；空值、`0` 或其他非法字符串都可能触发 `libgomp: Invalid value for environment variable OMP_NUM_THREADS`。

## 4. 统一变量与公共参数

为了让不同阶段尽量保持统一，建议先定义下面这些变量：

```bash
cd /root/SBBiShe

REPO_ROOT=/root/SBBiShe
DATA_ROOT="$REPO_ROOT/datasets"
EXP_ROOT="$REPO_ROOT/experiments"
REFLECTION_DIR="$DATA_ROOT/refool_reflections"

DEVICE=GPU
GPU_ID=0
SEED=666

BATCH_SIZE=32
NUM_WORKERS=8
TEST_INTERVAL=10
SAVE_INTERVAL=10

SMOKE_BATCH_SIZE=8
SMOKE_NUM_WORKERS=0

PR_LIST=(0.1 0.05 0.01)
```

说明：

- 对 CUB-200，建议服务器先从 `BATCH_SIZE=32` 起步；如果显存紧张，可继续调小。
- 本地 6GB 显卡做 smoke test 时，更建议使用 `SMOKE_BATCH_SIZE=8` 或更小。
- 如果只是验证代码链路，优先使用 `SMOKE_BATCH_SIZE / SMOKE_NUM_WORKERS`。

## 5. 阶段一：下载与整理数据集

### 5.1 一键自动下载并整理

如果机器上还没有 CUB-200，推荐直接执行：

```bash
python scripts/bootstrap_datasets.py --data-root "$DATA_ROOT" --dataset cub200
```

该命令会：

1. 下载原始 CUB-200 压缩包
2. 自动解压
3. 调用整理脚本转换成项目可直接读取的目录结构

整理后的目标目录为：

```text
datasets/CUB200/
|-- train/
`-- test/
```

### 5.2 指定 CUB 数据源

如果你已经手动下载好了 CUB 数据集，也可以显式指定来源：

#### 方式 A：使用本地压缩包

```bash
python scripts/bootstrap_datasets.py \
  --data-root "$DATA_ROOT" \
  --dataset cub200 \
  --cub-source /root/datasets/CUB_200_2011.tgz
```

#### 方式 B：使用已解压的原始目录

```bash
python scripts/bootstrap_datasets.py \
  --data-root "$DATA_ROOT" \
  --dataset cub200 \
  --cub-source /root/datasets/CUB_200_2011
```

### 5.3 单独执行整理脚本

如果你已经有原始 `CUB_200_2011` 目录，也可以直接运行整理脚本：

```bash
python scripts/prepare_cub200_dataset.py \
  --raw-root /root/datasets/CUB_200_2011 \
  --output-root "$DATA_ROOT/CUB200"
```

### 5.4 数据目录检查

整理完成后，至少应确认下面目录存在且非空：

- `datasets/CUB200/train`
- `datasets/CUB200/test`

CUB-200 不像 GTSRB 那样需要额外执行测试集平铺目录整理；只要 `train/test` 已经按类别分目录组织好，就可以直接训练。

## 6. 阶段二：检查 Refool 反射图目录

如果后续要跑 `Refool`，必须保证反射图目录存在：

- 本地：`datasets/refool_reflections`
- 服务器：`/root/SBBiShe/datasets/refool_reflections`

建议实验前确认目录中有可读图像文件。

如果只想先验证 `BadNets / Blended / WaNet`，这一步可以先跳过；但只要要跑 `Refool`，就必须提前准备。

## 7. 阶段三：1 epoch smoke test

建议先跑最短链路，确认：

- 数据读取正常
- CUB-200 目录结构正确
- 训练、REFINE、评估脚本都能跑通
- 输出目录能正常写入

推荐 smoke test 顺序：

1. `benign 1 epoch`（可选）
2. `BadNets 1 epoch`
3. `REFINE 1 epoch`
4. `eval_refine`

### 7.1 可选：先跑 benign 1 epoch

```bash
python scripts/train_benign.py \
  --dataset cub200 \
  --data-root "$DATA_ROOT" \
  --experiment-root "$EXP_ROOT" \
  --device "$DEVICE" \
  --gpu-id "$GPU_ID" \
  --epochs 1 \
  --disable-schedule \
  --batch-size "$SMOKE_BATCH_SIZE" \
  --num-workers "$SMOKE_NUM_WORKERS" \
  --seed "$SEED" \
  --test-interval 1 \
  --save-interval 1
```

说明：

- benign baseline 仍然复用统一训练入口，但现在会自动把投毒相关字段从 schedule 里剥离掉。
- 所以训练日志应该只反映干净训练，不再把 `poisoned_rate=0.05` 当成当前实验设置。
- 如果你之前看到 `poisoned_rate=0.05`，那只是旧版共享 schedule 的打印行为，不代表 benign 训练真的在做投毒。

### 7.2 最短攻击链路：BadNets 1 epoch

## 10. 正式运行时的建议：显式绑定 checkpoint

如果你只是顺序跑一遍，自动推断 checkpoint 通常够用。

但如果你要：

- 重复跑同一个投毒率
- 比较多组结果
- 避免不同时间的产物串组

那么建议在 REFINE 训练和评估阶段显式传入：

- `--attack-checkpoint`
- `--refine-checkpoint`
- `--arr-path`

### 10.1 REFINE 训练示例

```bash
python scripts/train_refine.py \
  --dataset cub200 \
  --attack badnets \
  --data-root "$DATA_ROOT" \
  --experiment-root "$EXP_ROOT" \
  --attack-checkpoint /root/SBBiShe/experiments/attacks/BadNets/<your_attack_dir>/ckpt_epoch_xx.pth \
  --device "$DEVICE" \
  --gpu-id "$GPU_ID" \
  --batch-size "$BATCH_SIZE" \
  --num-workers "$NUM_WORKERS" \
  --seed "$SEED" \
```

## 12. 建议的完整执行顺序

推荐按下面顺序执行：

1. 激活环境
2. 下载并整理 CUB-200
3. 检查 `datasets/CUB200/train` 与 `datasets/CUB200/test`
4. 检查 `datasets/refool_reflections`
5. 先跑 `BadNets -> REFINE -> eval` 的 1 epoch smoke test
6. 训练 benign
7. 训练四种攻击
8. 训练四种攻击对应的 REFINE
9. 评估四种攻击对应的 REFINE
10. 汇总最终结果

如果要做多投毒率对照，则把第 7、8、9 步按 `0.1 / 0.05 / 0.01` 三档展开执行。

## 13. 一句话原则

先用 1 epoch 确认 CUB-200 全链路可跑，再进入完整训练；正式结果优先保证 checkpoint 绑定清晰、不串组，再追求批量化执行。

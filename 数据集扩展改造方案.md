## 1. 文档目标

本文档用于指导当前仓库从 **GTSRB 专用实验链路** 扩展到：

- CUB-200-2011
- PACS

改造目标不是把仓库重写成通用研究框架，而是在**尽量不破坏现有论文主线**的前提下，为第二数据集验证提供一套可复用、可维护的实现方案。

当前仓库主线固定为：

- 数据集：GTSRB
- 分类模型：ResNet18
- 攻击：BadNets、Blended、WaNet、Refool
- 防御：REFINE

当前代码中与数据集强绑定的部分主要集中在 [scripts/_common.py](scripts/_common.py)、各个 `train_gtsrb_*.py` 脚本以及 [scripts/train_refine_gtsrb.py](scripts/train_refine_gtsrb.py)、[scripts/eval_refine_gtsrb.py](scripts/eval_refine_gtsrb.py)。

因此本次改造的核心原则是：

1. **先抽象数据集配置层**，再接入新数据集
2. **先做 CUB-200，再做 PACS**
3. **保留现有 GTSRB 脚本和实验口径**，避免影响论文主线复现
4. **新增通用入口脚本**，而不是继续按数据集复制脚本

补充：上述 1-4 在 CUB-200 阶段已完成；PACS 仍按本方案分阶段推进。

## 2. 当前代码中的数据集耦合点

在正式拆分 CUB-200 和 PACS 改造前，先明确当前代码里有哪些写死的假设。

### 2.1 [scripts/_common.py](scripts/_common.py) 中的固定假设

当前文件直接写死了：

- `NUM_CLASSES = 43`
- `IMAGE_SIZE = 32`
- `build_resnet18()` 默认构造 43 类分类器
- `load_gtsrb_datasets()` 默认读取 `datasets/GTSRB/train` 和 `datasets/GTSRB/testset`
- `make_gtsrb_transforms()` 默认按 GTSRB 的输入规则构造 transform
- `get_wanet_grid_paths()` / `infer_attack_checkpoint()` / `infer_refine_artifacts()` 默认实验名带 `gtsrb_`
- `default_refine_schedule()` 默认实验名前缀是 `gtsrb_refine_*`

### 2.2 各训练脚本中的固定假设

当前脚本命名和内部逻辑都直接绑定 GTSRB：

- [scripts/train_gtsrb_benign.py](scripts/train_gtsrb_benign.py)
- [scripts/train_gtsrb_badnets.py](scripts/train_gtsrb_badnets.py)
- [scripts/train_gtsrb_blended.py](scripts/train_gtsrb_blended.py)
- [scripts/train_gtsrb_wanet.py](scripts/train_gtsrb_wanet.py)
- [scripts/train_gtsrb_refool.py](scripts/train_gtsrb_refool.py)
- [scripts/train_refine_gtsrb.py](scripts/train_refine_gtsrb.py)
- [scripts/eval_refine_gtsrb.py](scripts/eval_refine_gtsrb.py)

这些脚本内部普遍直接调用：

- `load_gtsrb_datasets(...)`
- `build_resnet18()`
- 固定的 `gtsrb_*` 实验名

### 2.3 模型与防御层的可复用性

#### [core/models/resnet.py](core/models/resnet.py)
该文件本身是可复用的，因为：

- `ResNet(num, num_classes=10)` 已支持传入类别数

问题不在模型本体，而在调用层把类别数写死了。

#### [core/defenses/REFINE.py](core/defenses/REFINE.py)
该文件本身也具备扩展空间，因为：

- `REFINE(..., num_classes=10, ...)` 已支持传入类别数

问题同样主要在 [scripts/_common.py](scripts/_common.py) 的 `build_refine_defense()` 把 `num_classes` 固定成了 43。

### 2.4 攻击层的可复用性

[core/attacks/base.py](core/attacks/base.py) 当前支持的数据集类型包括：

- `DatasetFolder`
- `MNIST`
- `CIFAR10`

这意味着：

- 只要把 CUB-200 和 PACS 预处理成 `DatasetFolder` 风格
- 攻击实现大概率可以直接复用

因此，当前仓库最合理的扩展路径不是改攻击算法本体，而是把新数据集整理成当前攻击基类已经支持的输入格式。

# 第一章：接入 CUB-200 怎么改造

## 1.1 接入目标

CUB-200-2011 是细粒度分类数据集。接入它的意义主要是：

1. 验证 REFINE 在**细粒度类别区分**场景下是否仍有效
2. 让论文创新点从“交通标志”扩展到“细粒度自然图像分类”
3. 在不引入域协议复杂度的前提下，先把当前工程升级为“支持多个单域分类数据集”的实验仓库

因此，CUB-200 是比 PACS 更适合优先接入的数据集。

---

## 1.2 CUB-200 接入后的目标目录规范

推荐把 CUB-200 预处理成：

```text
datasets/CUB200/
|-- train/
|   |-- 001.Black_footed_Albatross/
|   |-- 002.Laysan_Albatross/
|   `-- ...
`-- test/
    |-- 001.Black_footed_Albatross/
    |-- 002.Laysan_Albatross/
    `-- ...
```

这样可以直接被 `torchvision.datasets.DatasetFolder` 读取，并复用当前攻击基类。

---

## 1.3 CUB-200 改造总原则

### 原则 1：先解耦数据集配置，再加脚本
不要先复制一套 `train_cub200_*.py`。应先把：

- 类别数
- 图像尺寸
- train/test 路径
- transform
- 实验名前缀

抽象成统一配置。

### 原则 2：保留现有 GTSRB 脚本
当前 GTSRB 主线已经比较稳定，因此建议：

- 原有 `train_gtsrb_*.py` 保留
- 另新增通用脚本，如 `train_attack.py`、`train_benign.py`、`train_refine.py`、`eval_refine.py`

当前状态更新：已按该原则落地，且 GTSRB 脚本已合并为薄包装以减少重复维护。

### 原则 3：CUB-200 先按普通单域分类接入
CUB-200 不需要额外 domain protocol，先按：

- clean train/test
- poisoned train/test
- REFINE clean training
- BA / ASR_NoTarget evaluation

直接复用当前主链路即可。

## 1.4 CUB-200 具体实施清单

### 1.4.1 改造 [scripts/_common.py](scripts/_common.py)

这是第一优先级文件。

#### 需要改造的方法 1：全局常量区

**当前问题**

当前存在：

```python
NUM_CLASSES = 43

DEFAULT_EXPERIMENT_ROOT = REPO_ROOT / "experiments"
```

`NUM_CLASSES` 和 `IMAGE_SIZE` 对 CUB-200 不成立。

**改造目标**

把固定常量改成“数据集配置表”。

**推荐改造方式**

新增：

```python
DATASET_SPECS = {
    "gtsrb": {
        "display_name": "GTSRB",

        "train_subdir": "GTSRB/train",
        "test_subdir": "GTSRB/testset",
        "task_type": "classification",

    "cub200": {
        "display_name": "CUB200",

        "image_size": 128,
        "train_subdir": "CUB200/train",
        "test_subdir": "CUB200/test",
        "task_type": "classification",
    },
}
```

**目的**

统一承载数据集差异，为模型构建、transform、路径推断、实验命名提供单一来源。

#### 需要新增的方法 2：`get_dataset_spec(dataset_name)`

**新增原因**

避免后面每个函数都自己写 `if dataset_name == ...`。

**推荐接口规范**

```python
def get_dataset_spec(dataset_name):
    """Return normalized dataset spec dict."""
```

**建议职责**

- 统一把数据集名转成小写
- 校验数据集是否受支持
- 返回对应配置字典

**目的**

后续所有构建逻辑都先取 `spec = get_dataset_spec(dataset_name)`。

---

#### 需要新增的方法 3：`add_dataset_args(parser)`

**新增原因**

当前 `parse_basic_args()` 只有：

- `--data-root`
- `--gpu-id`
- `--device`
- `--batch-size`
- `--num-workers`
- `--seed`
- `--experiment-root`

没有统一的 `--dataset` 参数。

**推荐接口规范**

```python
def add_dataset_args(parser):
    parser.add_argument(
        "--dataset",
        default="gtsrb",
        choices=("gtsrb", "cub200", "pacs"),
        help="Dataset name.",
    )
    return parser
```

**目的**

让后续新增通用脚本时，不再依赖脚本名表达数据集类型。

---

#### 需要改造的方法 4：`build_resnet18()`

**当前问题**

当前实现：

```python
def build_resnet18():
    return core.models.ResNet(18, NUM_CLASSES)
```

直接写死 43 类。

**推荐改造方式**

改成：

```python
def build_resnet18(num_classes):
    return core.models.ResNet(18, num_classes)
```

**为什么这样改**

`build_resnet18()` 应该是模型层接口，不应该再依赖某个特定数据集名称。

**目的**

让 GTSRB、CUB-200、PACS 都能复用统一模型构建逻辑。

---

#### 需要改造的方法 5：`make_gtsrb_transforms()`

**当前问题**

当前 transform 方法是按 GTSRB 命名且默认使用 `IMAGE_SIZE`。

**推荐改造方式**

把它拆成两个层次：

1. 通用入口：

```python
def make_dataset_transforms(dataset_name, attack_name=None, train=True, image_size=None):
    ...
```

2. 可选的具体实现：

```python
def make_gtsrb_transforms(attack_name=None, train=True, image_size=32):
    ...

def make_cub200_transforms(attack_name=None, train=True, image_size=128):
    ...
```

**CUB-200 推荐策略**

- train：`Resize -> RandomCrop/CenterCrop -> ToTensor`
- test：`Resize -> CenterCrop -> ToTensor`
- 若 `attack_name == "refool"`，保留和当前类似的标准化分支

**目的**

让输入预处理显式依赖数据集，而不是复用 GTSRB 的 32×32 流程。

---

# 第二章：接入 PACS 怎么改造

## 2.1 接入目标

PACS 不只是一个新的分类数据集，它还包含**域信息（domain）**。常见域包括：

- Photo
- Art painting
- Cartoon
- Sketch

因此 PACS 的价值不是单纯“新增一个 7 分类数据集”，而是为论文提供：

1. **跨域分布偏移**场景下的后门攻击验证
2. **跨域条件下** REFINE 的净化稳定性分析
3. 不同域之间 BA / ASR_NoTarget 差异的分析空间

也正因为 PACS 带有 domain protocol，它的接入复杂度明显高于 CUB-200。

当前状态更新：PACS 尚未进入完整代码落地阶段，本文第二章仍为实施方案。

---

## 2.2 PACS 改造建议分两阶段实施

### 阶段 A：先按普通分类数据集接入

目标是先跑通：

- benign
- attack
- refine
- eval

此时先不引入复杂域协议，可以采用：

- merged 方案
- 或单独选一个域（例如 `photo`）

### 阶段 B：再引入 domain-aware protocol

等阶段 A 跑通后，再支持：

- 多域训练
- 单域测试
- leave-one-domain-out
- per-domain BA / ASR_NoTarget 统计

推荐不要一开始就把两个阶段混在一起做，否则改造成本和调试成本都会明显上升。

---

## 2.3 PACS 接入后的推荐目录规范

### 方案 A：merged 普通分类格式

```text
datasets/PACS/merged/
|-- train/
|   |-- dog/
|   |-- elephant/
|   `-- ...
`-- test/
    |-- dog/
    |-- elephant/
    `-- ...
```

### 方案 B：按域组织的格式

```text
datasets/PACS/by_domain/
|-- photo/
|   |-- train/<class>/*
|   `-- test/<class>/*
|-- art_painting/
|   |-- train/<class>/*
|   `-- test/<class>/*
|-- cartoon/
|   |-- train/<class>/*
|   `-- test/<class>/*
`-- sketch/
    |-- train/<class>/*
    `-- test/<class>/*
```

推荐：

- **阶段 A** 先用 merged
- **阶段 B** 再使用 by_domain
